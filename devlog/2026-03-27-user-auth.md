# Email-Based User Authentication
**Date**: 2026-03-27  |  **Status**: Completed

## What Was Built
Email-based user authentication replacing the hardcoded `USERS` dictionary with a SQLite-backed `User` model, Flask-Login session management, and werkzeug password hashing. Provides login, registration, and logout flows via both HTML forms and JSON API endpoints. A `tier` column on the User model supports future subscription-based feature gating.

## Architecture
`functions/auth.py` defines a self-contained auth Blueprint (`auth_bp`) with a SQLAlchemy `User` model, Flask-Login integration, and manual CSRF protection. `app.py` calls `init_auth(app)` to wire up SQLite (`instance/dream_chat.db`), register the blueprint, and seed a demo user on first run. Protected routes use `@login_required` or check `current_user.is_authenticated`. The `session["username"]` key is set on login for backward compatibility with existing code that reads it.

## Key Files
| File | Purpose |
|------|---------|
| `functions/auth.py` | User model, login/register/logout routes, JSON API, CSRF helpers, `init_auth()` |
| `app.py` | SECRET_KEY from env (fallback to random), `init_auth()` call, `@login_required` on routes |
| `templates/login.html` | Login form with CSRF token and flash messages |
| `templates/register.html` | Registration form with email, password, confirm password, CSRF |
| `tests/test_auth.py` | 63 tests across 9 test classes |

## Technical Decisions
- **SECRET_KEY from env, not hardcoded** -- `os.environ.get("SECRET_KEY") or secrets.token_hex(32)` avoids shipping a secret in source while still working locally without config.
- **Manual CSRF over Flask-WTF** -- a per-session `secrets.token_hex(32)` stored in `session["_csrf_token"]` and validated on POST. Keeps dependencies minimal; form templates include a hidden `_csrf_token` field.
- **SQLAlchemy with SQLite** -- simple single-file persistence via `instance/dream_chat.db`; `init_auth` calls `db.create_all()` so no migration tooling is needed yet.
- **JSON API endpoints** (`/api/login`, `/api/logout`) -- preserves backward compatibility for AJAX callers alongside the form-based flow.
- **Email normalization** -- `strip().lower()` on all email inputs ensures case-insensitive matching.

## Usage
```bash
# Set a stable secret key for production
export SECRET_KEY="your-production-secret"

# Run the app (demo user seeded automatically: demo@example.com / demo123)
python app.py

# Visit /login or /register in browser
```

## Testing
```bash
pytest tests/test_auth.py -v   # 63 tests
```
Covers: registration validation (duplicates, short passwords, mismatched confirm), login success/failure, logout session clearing, auth enforcement on protected routes, JSON API endpoints (including content-type validation), CSRF rejection, SQL injection resilience, email normalization, unicode passwords, User model constraints.

## Known Limitations
- **No password reset flow** -- users cannot recover accounts if they forget their password.
- **No email verification** -- registration does not confirm email ownership.
- **No rate limiting** -- login/register endpoints are vulnerable to brute force without an external rate limiter.
- **SQLite single-writer** -- adequate for single-server use but will need migration to PostgreSQL for concurrent multi-process deployments.
- **`tier` column unused** -- the subscription tier field is in place but no code gates features on it yet.
